home-kai

Своя оверлей-сеть в духе Tailscale: все машины доступны по адресам 100.87.x.x, при этом ни на одной из них не открыт ни один входящий порт. Data plane — WireGuard, control plane — свой координатор на Go.

Как это работает

  1. Координатор — единственный «мозг»: хранит публичные ключи узлов, выдаёт оверлей-адреса и раздаёт каждому узлу его netmap — полный список WireGuard-пиров, которые нужно установить.
  2. Агент на каждом узле держит WireGuard-интерфейс (kai0 на Linux, utunN на macOS) и синхронизирует пиров через HTTP long-poll. Все соединения агента — исходящие: HTTPS к координатору и UDP к хабу.
  3. Хаб (VPS с белым IP) — WireGuard-релей: узлы за NAT дозваниваются до него, он форвардит пакеты между ними (ip_forward). Только у него открыты порты наружу.

Маршрутизация — чистый WireGuard cryptokey routing: у спока один пир (хаб) с AllowedIPs = 100.87.0.0/16, у хаба — каждый узел по /32. PersistentKeepalive = 25 держит NAT-мэппинги живыми.

Координатор может лежать — сеть работает. Netmap кэшируется в state-файле агента и применяется при старте; туннели живут независимо от control plane. Координатор нужен только чтобы добавлять/удалять узлы.

Безопасность

  • Приватный ключ WireGuard генерируется на узле и никогда его не покидает (исключение — static peers, см. ниже).
  • Enrollment — по одноразовому токену с TTL (по умолчанию 1 час); после регистрации узел аутентифицируется собственным секретом (на сервере хранится только hash).
  • TLS координатора — self-signed сертификат, агент пинит его sha256-отпечаток (передаётся вместе с токеном). Не нужны ни домен, ни Let’s Encrypt, ни доверие системным CA; смена IP сервера ничего не ломает.

Установка с нуля (VPS)

# 1. бинари и юниты (с рабочей машины)
just provision-vps && just deploy-vps

# 2. на VPS: админ-токен и конфиг
kai-coordinator gen-admin-token > /root/kai-admin-token.txt && chmod 600 /root/kai-admin-token.txt
cat > /etc/kai/coordinator.toml <<'EOF'
listen           = ":8443"
public_url       = "https://vpn.example.com:8443"
hub_endpoint     = "vpn.example.com:51820"
overlay_cidr     = "100.87.0.0/16"
data_dir         = "/var/lib/kai"
admin_token_hash = ""
EOF
useradd -r -s /usr/sbin/nologin kai
chown root:kai /etc/kai/coordinator.toml && chmod 640 /etc/kai/coordinator.toml
systemctl enable --now kai-coordinator
journalctl -u kai-coordinator | grep fingerprint   # отпечаток — понадобится всем агентам

# 3. хаб — это тоже агент на том же VPS
home-kai token create --name vps-hub          # env: KAI_URL / KAI_ADMIN_TOKEN / KAI_FINGERPRINT
kai-agent up --coordinator https://127.0.0.1:8443 --token ... --fingerprint ... --hub
# ^ после первого enroll достаточно: systemctl enable --now kai-agent

Подключение устройств

Linux-сервер или Mac

  1. Создай токен: home-kai token create --name home-server — команда сразу напечатает готовую join-строку.
  2. На устройстве (root обязателен — агент управляет сетевым интерфейсом): 
    sudo kai-agent up \
  --coordinator https://vpn.example.com:8443 \
  --token <одноразовый токен> \
  --fingerprint <отпечаток>
  3. Постоянный запуск: Linux — deploy/systemd/kai-agent.service, macOS — deploy/launchd/dev.kai.agent.plist (в /Library/LaunchDaemons). Повторный kai-agent up без флагов использует сохранённый state (/var/lib/kai-agent/state.json).

iPhone / Android / роутер (static peer)

Свой агент туда не поставить, поэтому используется официальное WireGuard-приложение:

home-kai peer create iphone          # напечатает конфиг + QR прямо в терминал
home-kai peer create iphone --png iphone.png   # и/или в PNG

Сканируешь QR в приложении WireGuard — устройство получает адрес и ходит во всю сеть через хаб. Для роутера/NAS используй сам .conf. В конфиге прописан DNS хаба, так что имена вида nas.kai (и просто nas — через search-домен) работают и с телефона.

Для static peers ключевая пара генерируется на сервере (приватный ключ обязан попасть в конфиг/QR) — это единственное место, где приватник существует вне устройства. Такие пиры всегда ходят через хаб.

Администрирование

Web UI

Панель управления встроена в координатор: https://vpn.example.com:8444/ui. Вход по admin-токену (/root/kai-admin-token.txt на VPS) — токен обменивается на HttpOnly session-cookie на 30 дней и в браузере не хранится; сессии живут в памяти координатора, его рестарт разлогинивает. Панель показывает устройства с online-статусом (теги, кнопка одобрения, включение анонсированных подсетей), static peers (конфиг/QR, тумблер полного туннеля), публикации сервисов, ACL-политики, индикатор network lock и журнал последних событий; умеет создавать enroll-токены (готовая join-команда) и удалять узлы. Таблицы перерисовываются только при изменении данных — клики и открытые модалки не сбиваются.

На порту 8444 — Let's Encrypt сертификат, выпущенный прямо на IP-адрес (профиль shortlived, 160 часов): браузер доверяет ему без предупреждений. certbot на VPS продлевает серт по systemd-таймеру, deploy-hook копирует файлы в /etc/kai/ui-tls/, а координатор перечитывает их с диска без рестарта. Агенты по-прежнему ходят на 8443 с self-signed сертификатом и пиннингом отпечатка — LE тут не годится: его отпечаток менялся бы каждые 6 дней. :8443/ui работает как fallback (с предупреждением браузера).

export KAI_ADMIN_TOKEN=...     # /root/kai-admin-token.txt на VPS
export KAI_FINGERPRINT=...     # из лога координатора

Прямые p2p-соединения

Споки пробуют соединиться друг с другом напрямую, минуя хаб. Кандидаты — LAN-адреса (агент сам сообщает) и публичные endpoint’ы, которые хаб наблюдает по keepalive. Пробинг не рвёт связь: пир ставится с пустыми AllowedIPs (handshake ходит, трафик — нет), и только подтверждённый путь получает /32, который перебивает /16 хаба. Стух handshake >3 минут — мгновенный откат на relay и повторная проба позже. Смотреть: home-kai status (колонка PATH).

Subnet router: доступ к чужой LAN

sudo kai-agent up --advertise-routes 192.168.1.0/24   # на узле в этой LAN
home-kai node routes <node_id> --enable 192.168.1.0/24     # разрешение админа (или чекбокс в UI)

После включения все устройства достают эту подсеть через оверлей (телефонам с готовым конфигом надо перегенерировать conf). Узел-роутер сам включает форвардинг и маскарадит трафик в LAN. Если подсеть совпадает с локальной сетью другого узла (типично: docker-бриджи 172.17–19.0.0/16) — тот узел пропустит маршрут и напишет об этом в лог, его локальная сеть не пострадает.

Публикации (funnel)

Кнопка «Опубликовать сервис» в UI: публичный порт VPS → сервис в оверлее (nas.kai:8096 или 100.87.0.4:8096). Хаб просто форвардит TCP — TLS и аутентификация остаются на самом сервисе. Порт открыт всему интернету.

ACL-политики (сегментация)

По умолчанию оверлей плоский — все видят всех. Чтобы ограничить доступ, назначь устройствам теги (home-kai node tag <id> --tags web, home-kai peer tag <id> --tags phones или кнопка «теги» в панели) и заведи политику: home-kai policy create имя --from admin --to web --proto tcp --ports 22,443. Пустой --from/--to означает «любой».

Как только появляется первая включённая политика, сеть переключается в режим «запрещено всё, кроме явно разрешённого». Enforcement двухуровневый. Координатор вычисляет для каждого узла список разрешённого входящего трафика и раздаёт в netmap; Linux-агент применяет его iptables-цепочкой KAI-FILTER на входе интерфейса kai0 (физический SSH и управление не затрагиваются — они не идут через оверлей). Дополнительно хаб и subnet-роутеры фильтруют пересылаемый трафик цепочкой KAI-FORWARD: под ACL попадают и static peers (телефоны), и LAN-подсети за роутером — подсеть наследует теги анонсирующего узла. macOS-узлы локальный фильтр не ставят (нет iptables), поэтому при включённом ACL координатор не раздаёт их p2p-кандидатов: весь их трафик идёт через хаб и фильтруется там.

Фильтр контролирует и ICMP: если в политике только tcp/443, пинг между этими узлами тоже не пройдёт — это ожидаемо. Заведи политику с протоколом «любой» или icmp, если нужен ping.

Одобрение узлов (peer approval)

С require_approval = true в конфиге новый узел после enroll висит в состоянии «ожидает»: он никого не видит и невидим остальным, пока админ не нажмёт «одобрить» в панели или не выполнит home-kai node approve <id>. Хаб доверяется автоматически.

Журнал событий

Ключевые действия (подключение/одобрение/удаление узлов, теги, политики, публикации, lock, вход в панель) пишутся в журнал — карточка «Журнал» в UI, home-kai events или GET /v1/admin/events. Если задан event_webhook, на каждое событие уходит JSON-POST — можно завести мост в SIEM или чат.

Network lock

Защита от компрометации координатора: home-kai lock init создаёт ed25519-ключ на админской машине (~/.config/kai/lock.key — забэкапь, второй копии нет), home-kai lock sign подписывает привязку (WG-ключ, overlay-IP) каждого устройства. Агенты пинят публичный ключ и молча отбрасывают неподписанных пиров — координатор, даже взломанный, не может добавить в сеть свой узел. Каждое новое устройство (и каждая ротация ключа) требует home-kai lock sign — до этого узел не увидит сеть.

Ротация ключей

--rekey-days 30 у агента — авторотация: новый ключ регистрируется у координатора и подменяется на живом интерфейсе, пики применяют новый netmap за секунды. Прерванная ротация самолечится при старте (агент ре-ассертит свой ключ). При включённом lock авторотация отключена — ротируй вручную и подписывай.

DNS-имена устройств (опционально)

Если в конфиге координатора включена секция [dns] (провайдер Timeweb), каждое новое устройство при регистрации получает публичную A-запись {16 случайных символов}.{домен} → его оверлей-IP. Запись резолвится откуда угодно, но IP из 100.87.0.0/16 достижим только изнутри сети.

Некоторые домашние роутеры с DNS rebind protection режут ответы с приватными/CGNAT адресами — если имя не резолвится дома, добавь исключение для своего домена в настройках роутера.

Диагностика

СимптомПричина и что делать
Первый ping после подключения теряется Норма: первый пакет инициирует WireGuard-handshake.
Mac не отвечает на ping по оверлею, хотя сам всех пингует Stealth mode файрвола macOS молча жрёт ICMP. Туннель жив — проверяй TCP-ом (nc, ssh, curl).
SSH подключается и виснет; мелкие пинги ходят, большие пакеты — нет MTU. По умолчанию 1420; за PPPoE/LTE поставь 1400/1380 (поле mtu в state-файле агента, перезапусти агент).
Узел «пропал» из сети home-kai node list — смотри last seen. На узле: kai-agent status, лог агента. Агент переподключается сам с экспоненциальным backoff.
Координатор упал Существующие туннели работают. Новые узлы добавить нельзя до восстановления. systemctl restart kai-coordinator на VPS.
У VPS сменился публичный IP Агенты получат новый endpoint хаба из netmap (адрес координатора в их конфиге поправь на новый). Конфиги static peers содержат IP статически — перегенерируй (home-kai peer create) или используй DNS-имя в hub_endpoint.
Ничего не работает вообще Проверь cloud-firewall провайдера: должны быть открыты TCP 8443 и UDP 51820 на VPS.

Под капотом: протокол

  1. Enroll: агент генерирует ключи → POST /v1/enroll с одноразовым токеном и своим публичным ключом → координатор выдаёт node_id, персональный секрет, оверлей-IP → агент сохраняет state (0600).
  2. Sync: GET /v1/netmap?since=N — сервер держит запрос до 55 секунд; отвечает сразу, как только версия netmap вырастет (кто-то добавился/удалился). Netmap всегда полный, применение идемпотентно (ReplacePeers).
  3. Status: раз в минуту (и сразу после старта) агент шлёт heartbeat: LAN-адреса, анонсируемые подсети; хаб дополнительно сообщает наблюдаемые публичные endpoint’ы пиров — из них координатор собирает кандидатов для p2p-пробинга.
  4. Rekey: POST /v1/rekey — идемпотентная замена WG-ключа узла.

Roadmap

  • M1 — hub-and-spoke: готово, проверено (включая работу data plane при выключенном координаторе).
  • M2 — static peers + QR: код готов; осталось проверить на реальном iPhone.
  • M3 — прямые соединения: готово (prober с бесшовным fallback на хаб); проверено на паре macOS ↔ Linux-узел.
  • M4 — качество жизни: готово — /etc/hosts-имена, home-kai status/ping, subnet router, exit node, публикации, ротация ключей, network lock, /metrics.
  • M5 — управление доступом: готово — ACL-политики с тегами, одобрение узлов (peer approval), журнал событий с webhook.
  • Осталось: launchd/автозапуск на Mac, kai netcheck (классификация NAT), публичные DNS-имена через Timeweb (код готов, ждёт домена).
